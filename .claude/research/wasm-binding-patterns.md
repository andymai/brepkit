## 1. OpenCascade.js

**Repository:** https://github.com/donalffons/opencascade.js (beta branch has latest)

### Architecture: Code-Generated Emscripten Bindings

OpenCascade.js does NOT use hand-written bindings. It auto-generates everything from
C++ headers using a multi-stage Python pipeline:

**File organization** (from `src/` on beta branch):
```
src/
  generateBindings.py     # Main AST-driven binding generator
  bindings.py             # Binding support utilities
  compileBindings.py      # Compilation of generated bindings
  buildFromYaml.py        # YAML-configured custom builds
  Common.py               # Shared utilities
  wasmGenerator/          # WASM-specific code generators
  filter/                 # C++ declaration filtering
  patches/                # Manual patches for edge cases
```

### Binding Generation Pattern

The core pattern is **AST parsing via libclang** -> **parallel code generation**:

1. Parse C++ headers into an AST using Python's libclang bindings
2. Filter declarations into classes, templates, and enums via three filter functions
3. For production builds, divide work across CPU cores with `multiprocessing.Pool`
4. Generate **two outputs per declaration**:
   - Emscripten embind C++ code (`.cpp`)
   - TypeScript definitions (`.d.ts.json`)

**Key design decision:** They chose Embind over WebIDL specifically to preserve
C++ comments and type information in TypeScript output.

Source: https://github.com/donalffons/opencascade.js/discussions/27

### Error Handling

No structured error propagation from C++ to JS. Operations that fail in OCCT
typically throw C++ exceptions which Emscripten converts to JS exceptions.
The TypeScript types provide some compile-time safety but runtime errors are
unstructured strings.

### TypeScript Types

Generated automatically alongside bindings. Types are stored as `.d.ts.json`
intermediate format and assembled into final `.d.ts` files. This is important:
types and bindings come from the SAME code generation pass, ensuring they
never drift.

### Batch Operations

OpenCascade.js does NOT have a batch API. Each OCCT operation is a separate
JS function call crossing the WASM boundary. The underlying OCCT library
does support multi-argument boolean operations (`BRepAlgoAPI_BooleanOperation`
with `SetArguments`/`SetTools`), but the JS binding layer doesn't expose a
generic batch dispatch mechanism.

Source: https://dev.opencascade.org/content/boolean-operations-multiple-arguments

### What brepkit Could Adopt

- **Code generation for TypeScript types from Rust types** -- generating .d.ts
  from Rust struct definitions ensures types never drift. Currently brepkit's
  TypeScript types are generated by wasm-bindgen, which works but produces
  minimal types.
- **Custom build system** -- OpenCascade.js lets users include only the OCCT
  classes they need, dramatically reducing bundle size. brepkit could offer
  feature-gated WASM modules.

### What Doesn't Apply

- The entire Emscripten/C++ stack is irrelevant -- brepkit uses wasm-bindgen.
- The Python code generation pipeline is overkill for a Rust project where
  proc macros can do the same work at compile time.
- The lack of batch operations is a weakness, not a pattern to follow.

## 2. Manifold

**Repository:** https://github.com/elalish/manifold
**npm:** manifold-3d

### Architecture: Three-Layer Binding (C++ -> Embind -> JS Wrapper)

**File organization** (`bindings/wasm/`):
```
bindings/wasm/
  bindings.cpp          # Emscripten EMSCRIPTEN_BINDINGS macro registrations
  helpers.cpp           # Data conversion functions (MeshGL <-> JS)
  bindings.js           # JavaScript wrapper layer with idiomatic API
  types/
    manifold-encapsulated-types.d.ts
    manifold-global-types.d.ts
    manifold-root.d.ts
  examples/
  test/
  lib/
```

### Mesh Data Transfer Pattern (CRITICAL for brepkit)

Manifold uses `MeshGL` -- a flat-buffer structure optimized for GPU upload:

```cpp
// helpers.cpp -- C++ to JS conversion
MeshGL2JS(const MeshGL& mesh) {
    // Uses typed_memory_view to create JS TypedArray VIEW of C++ vector data
    // Then calls .slice() to copy across the WASM boundary
    // Each array field transferred individually
}

// JS to C++ conversion
MeshJS2GL(val mesh_js) {
    // Uses convertJSArrayToNumberVector<T>() to parse JS arrays
    // Checks for optional fields using val::undefined()
}
```

**MeshGL structure fields:**
- `triVerts` (triangle indices, `uint32_t[]`)
- `vertProperties` (interleaved vertex attributes, `float[]`)
- `mergeFromVert`, `mergeToVert` (vertex correspondence for manifold reconstruction)
- `runIndex`, `runOriginalID`, `faceID` (run-length encoding for face groups)
- `halfedgeTangent`, `runTransform` (normals/tangent data)
- `numProp`, `tolerance` (metadata)

**Key insight:** All vertex properties are interleaved into a SINGLE buffer with
vertices duplicated as necessary. This means one `TypedArray.slice()` call per
field rather than one per vertex.

Source: https://deepwiki.com/elalish/manifold

### Error Handling: Status Codes (NOT Exceptions)

Manifold uses an enum-based error status rather than throwing:

```cpp
// bindings.cpp
enum_<Manifold::Error>("status")
  .value("NoError", Manifold::Error::NoError)
  .value("NonFiniteVertex", Manifold::Error::NonFiniteVertex)
  .value("NotManifold", Manifold::Error::NotManifold)
  .value("VertexOutOfBounds", Manifold::Error::VertexOutOfBounds)
  ...
```

```js
// bindings.js
// Check status after construction
const status = manifold.status();
if (status !== 'NoError') {
    throw new Module.ManifoldError(status);
}
```

**Design principle:** Invalid inputs produce an empty Manifold with a queryable
error status. Subsequent operations on invalid manifolds PROPAGATE the error
status automatically. This is a monadic error pattern -- the error flows through
without each caller checking.

### JS Wrapper Layer Pattern

`bindings.js` adds idiomatic JavaScript on top of raw Embind:
- C++ methods use underscore prefixes (`_add`, `_Translate`)
- JS wrapper provides clean names and does type conversion
- Memory management helpers: `addFunction`/`removeFunction` for callbacks
- `Module.setup()` with initialization guard prevents double-init

### What brepkit Could Adopt

- **Packed buffer transfer pattern.** brepkit already does this with
  `JsMesh::packed_buffer()` -- good. Manifold validates this approach.
- **Error status propagation** rather than exception-heavy patterns. brepkit
  could return error status on solids so downstream operations don't need
  to re-validate.
- **Interleaved vertex properties** in a single buffer. brepkit currently
  keeps positions, normals, and indices separate.

### What Doesn't Apply

- Emscripten/C++ binding layer not applicable.
- `MeshGL` merge vectors for manifold reconstruction -- brepkit uses B-Rep
  topology directly, not mesh topology.
- Web Worker integration is relevant but orthogonal to binding architecture.

## 3. truck

**Repository:** https://github.com/ricosjp/truck
**WASM crate:** `truck-js/` (not published to crates.io)

### Architecture: Wrapper Structs via `IntoWasm` Trait

**File organization** (`truck-js/src/`):
```
truck-js/src/
  lib.rs       # IntoWasm trait, module declarations, lint config
  shape.rs     # Topology type wrappers (Vertex, Edge, Wire, Face, Shell, Solid)
  builder.rs   # Primitive construction WASM bindings
  shapeops.rs  # Boolean operation WASM bindings
  polygon.rs   # Mesh/polygon WASM bindings
  step_in.rs   # STEP I/O WASM bindings
```

**Dependencies:**
```toml
wasm-bindgen = "0.2.114"
js-sys = "0.3.91"
gloo = "0.11.0"          # Browser console logging for errors
derive_more = "2.1.1"     # From, Into, Deref, DerefMut, AsRef derives
serde_json = "1.0.149"
# Local path deps:
truck-meshalgo, truck-modeling, truck-shapeops, truck-stepio, truck-topology
```

### The `IntoWasm` Trait Pattern

truck-js defines a conversion trait that all topology types implement:

```rust
// lib.rs (reconstructed from summaries)
pub trait IntoWasm {
    type WasmWrapper: From<Self>;
    fn into_wasm(self) -> Self::WasmWrapper {
        self.into()
    }
}
```

### Topology Wrapper Macro: `toporedef!`

The key pattern -- a macro generates wrapper structs for each topology type:

```rust
// shape.rs (reconstructed from summaries)
// Each topology type gets:
#[wasm_bindgen]
#[derive(Clone, Debug, From, Into, Deref, DerefMut, AsRef)]
pub struct Vertex(truck_modeling::Vertex);
// Repeat for Edge, Wire, Face, Shell, Solid

// Plus an AbstractShape enum for polymorphic dispatch:
pub enum AbstractShape {
    Vertex(Vertex),
    Edge(Edge),
    Wire(Wire),
    Face(Face),
    Shell(Shell),
    Solid(Solid),
}
```

Each wrapper provides:
1. Upcast to `AbstractShape` (for polymorphic JS APIs)
2. Type-checking methods (`is_vertex()`, `is_edge()`, etc.)
3. Downcast methods (`into_vertex()`, `as_vertex()`)
4. JSON serialization via `serde_json` (`from_json`, `to_json`)

### Error Handling: Option + Console Logging

truck-js uses a deliberately simple pattern:

```rust
// Fallible operations return Option, log errors to browser console
pub fn some_operation(&self) -> Option<Face> {
    match self.0.try_something() {
        Ok(result) => Some(result.into_wasm()),
        Err(e) => {
            gloo::console::error!(format!("{e}"));
            None
        }
    }
}
```

This is DIFFERENT from brepkit's approach (which uses `Result<T, JsError>`).
Returning `Option` loses error details for the JS caller.

### Builder Operations: Macro-Based Polymorphism

Two macros handle the "one operation, many shape types" problem:

```rust
// builder.rs (reconstructed)
// intopt! -- convert flat f64 arrays to 3D types
intopt!(Point3, origin, Vector3, axis);

// derive_all_shape! -- dispatch operation across shape variants
// Uses nested if-let chains testing each variant
derive_all_shape!(
    fn translate(shape: &AbstractShape, dx: f64, dy: f64, dz: f64) -> AbstractShape
    // Expands to: if let Some(v) = shape.as_vertex() { ... }
    //             else if let Some(e) = shape.as_edge() { ... }
    //             ...
);
```

### Boolean Operations Pattern

```rust
// shapeops.rs (reconstructed)
#[wasm_bindgen]
pub fn and(solid_a: &Solid, solid_b: &Solid, tol: Option<f64>) -> Option<Solid> {
    let tol = tol.unwrap_or(SHAPEOPS_TOLERANCE);  // default: 0.05
    shapeops::and(&solid_a.0, &solid_b.0, tol)
        .map(|s| s.into_wasm())
}
```

Note: truck exposes booleans as **free functions**, not methods on a kernel.
Each solid is a self-contained value, not an arena handle.

### What brepkit Could Adopt

- **`IntoWasm` trait pattern.** Clean separation between internal types and
  WASM wrappers. brepkit could define `trait IntoJs` with associated types
  for each topology entity.
- **`derive_more` for wrapper boilerplate.** `From`, `Into`, `Deref` derives
  eliminate manual conversion code.
- **Module-per-domain organization.** shape.rs, builder.rs, shapeops.rs is
  a much cleaner split than brepkit's 7800+ line kernel.rs monolith.

### What Doesn't Apply

- truck uses **value-based topology** (each Solid is an owned value), while
  brepkit uses **arena-based topology** (SolidId handles into a shared arena).
  This means truck can pass `&Solid` across FFI; brepkit must pass `u32` handles.
- `Option`-based error handling loses too much information. brepkit's `Result`
  approach is better.
- truck's `AbstractShape` polymorphism is needed because truck exposes each
  topology type as a separate WASM class. brepkit uses a single `BrepKernel`
  with u32 handles, making polymorphism unnecessary at the WASM boundary.

## 4. wasm-bindgen Best Practices

### Error Handling in WASM

**`catch_unwind` is unreliable on wasm32-unknown-unknown.**

`std::panic::catch_unwind` has no effect when `panic=abort` is set, which is
the default for `wasm32-unknown-unknown`. Even when `panic=unwind` is
configured, catching a panic in WASM leaves the instance in an inconsistent
state -- "the safest course of action after catching an exception is to
discard the WASM instance and create a new one from scratch."

Source: https://users.rust-lang.org/t/catch-panic-in-wasm/57569

**brepkit currently uses `catch_unwind` in fillet operations.** This is
dangerous. With the default `panic=abort` on wasm32, `catch_unwind` is a
no-op and the panic will abort the entire WASM module. The code at
kernel.rs:641 and kernel.rs:6457 needs to be reconsidered.

**Recommended pattern:** Use `console_error_panic_hook` for debugging panics,
but never rely on catching them. Validate inputs thoroughly before calling
operations that might panic.

Source: https://github.com/rustwasm/console_error_panic_hook

**Result<T, JsValue> is the only supported error return type.** wasm-bindgen
converts `Result<T, E>` where `E: Into<JsValue>` into a JS throw. brepkit's
`WasmError` already implements `Display` which converts to `JsError` which
implements `Into<JsValue>`. This is the correct pattern.

Source: https://alexcrichton.github.io/wasm-bindgen/reference/types/result.html

### Organizing Large APIs (100+ Functions)

**`js_namespace` on exports** (added in recent wasm-bindgen versions):

The `js_namespace` attribute can now be used on exported types, not just
imports. This allows organizing exported structs into nested namespaces:

```rust
#[wasm_bindgen(js_namespace = "Modeling")]
pub struct BrepKernel { ... }

// Accessed in JS as: new Modeling.BrepKernel()
```

Combined with the fix for same `js_name` but different `js_namespace` values,
this enables namespaced API organization without symbol collisions.

Source: https://github.com/wasm-bindgen/wasm-bindgen/releases

**Private types for internal use:**

Recent wasm-bindgen adds a `private` attribute on exported types to generate
internal export types that aren't part of the public JS interface but can be
used as function arguments/return types.

**Multiple `impl` blocks are supported** and sometimes required. brepkit
already uses 5 `impl BrepKernel` blocks (lines 177, 4140, 4216, 4716, 6067).
This is fine and expected for large APIs.

### Bulk Data Transfer Performance

Three approaches compared:

| Approach | Speed | Code Size | Notes |
|----------|-------|-----------|-------|
| `serde-wasm-bindgen` | Fastest for most cases | Smallest | Official recommendation |
| JSON serialization | 0.5x - 2x vs serde-wasm-bindgen | Larger | Browsers' JSON parsers are fast |
| Raw `Vec<u8>` packed buffer | Fastest for bulk numeric data | Manual | One copy across WASM boundary |

Source: https://rustwasm.github.io/docs/wasm-bindgen/reference/arbitrary-data-with-serde.html
Source: https://neugierig.org/software/blog/2024/04/rust-wasm-to-js.html

**brepkit's packed buffer approach for mesh data is optimal.** The
`JsMesh::packed_buffer()` method returns a single `Vec<u8>` that JS unpacks
into TypedArrays. This avoids three separate `.clone()` + FFI copies that
the individual getters would incur. This is the same approach Manifold uses
(`typed_memory_view` + `.slice()`).

For structured data (operation results, error details), `serde-wasm-bindgen`
is preferred over JSON. brepkit currently uses JSON strings for batch results
(`execute_batch` returns a JSON string). This adds a serialize-to-JSON step
on Rust side and a parse step on JS side.

### tsify / tsify-next for TypeScript Generation

**Original tsify is unmaintained.** Use `tsify-next` (fork by AmbientRun):
https://github.com/AmbientRun/tsify-next

**Known caveats:**
1. Won't generate `.d.ts` if a crate has exactly one tsify type with
   `into_wasm_abi` but not `from_wasm_abi` and the type only appears in
   indirect return values. Workaround: `codegen-units = 1` or add a dummy
   direct return function.
   Source: https://github.com/madonoharu/tsify/issues/50

2. `Vec<T>` where T is tsified does NOT work -- `JsObject` trait is not
   implemented. Must wrap in a custom struct or serialize to JSON.
   Source: https://github.com/rustwasm/wasm-bindgen/issues/3692

3. Two serialization modes: `json` (default, via serde_json) and `js`
   (via serde-wasm-bindgen, planned to become default).

**For brepkit:** tsify-next would be useful for typed batch operation
results, typed error responses, and complex query results. Not needed for
the simple u32 handle returns that dominate the API.

## 5. Proc Macro Patterns for WASM

### Existing Crates

**`wasm-bindgen-struct`** (https://crates.io/crates/wasm-bindgen-struct):
- Generates getters/setters for struct fields with automatic camelCase conversion
- Handles async patterns via `MapValue<T, U>` type
- Designed for IMPORTING JS types into Rust, not for reducing EXPORT boilerplate
- **Not useful for brepkit's case** (brepkit exports Rust types to JS)

**`wasm_refgen`** (https://crates.io/crates/wasm_refgen):
- Generates boilerplate for upcasting JS duck-typed references to concrete Rust types
- Placed above `#[wasm_bindgen]` on impl blocks
- Cheap clone (copies a lightweight JS pointer, not the data)
- **Not useful for brepkit** (brepkit doesn't receive typed JS objects back)

### Auto-Registration: inventory vs linkme

**Neither works reliably on wasm32-unknown-unknown.**

- **linkme**: Uses linker sections. WASM support was closed as "not actionable"
  in Jan 2023. Custom data sections exist in WASM but cross-crate sections
  get discarded due to linker bugs.
  Source: https://github.com/dtolnay/linkme/issues/6

- **inventory**: Uses global constructors (code before `main()`). This
  mechanism doesn't exist in wasm32-unknown-unknown.

**Implication for brepkit:** Auto-registration of batch dispatch operations
via these crates is NOT possible on WASM targets. The dispatch table must be
built explicitly (as it currently is in `dispatch_op`).

### Viable Proc Macro Approach for brepkit

Since neither inventory nor linkme works on WASM, and wasm-bindgen itself
is a proc macro, the viable approach is a **custom proc macro that generates
both the wasm-bindgen export AND the batch dispatch arm** from a single
source of truth.

Hypothetical design:

```rust
// In a proc-macro crate (e.g., brepkit-wasm-macros)
#[proc_macro_attribute]
pub fn wasm_op(attr: TokenStream, item: TokenStream) -> TokenStream {
    // Parse the function
    // Generate:
    // 1. The #[wasm_bindgen] impl method (direct call path)
    // 2. A const entry for a dispatch table
    // 3. Parameter extraction from serde_json::Value (batch path)
}

// Usage:
#[wasm_op(name = "makeBox", batch_args = {width: f64, height: f64, depth: f64})]
pub fn make_box(&mut self, width: f64, height: f64, depth: f64) -> Result<u32, JsError> {
    // ...
}

// Expands to both the wasm_bindgen method AND the dispatch_op match arm
```

**Challenge:** `#[wasm_bindgen]` is itself a proc macro, and Rust proc macros
cannot compose arbitrarily. The custom macro would need to emit the
`#[wasm_bindgen]` attribute in its output token stream, which IS supported
(proc macros can emit other proc macro attributes).

There is precedent for this pattern: the wasm-bindgen issue #2758 describes
a prototype proc macro that auto-generates import bindings from annotated
impl blocks using `syn` + `quote`.

Source: https://github.com/rustwasm/wasm-bindgen/issues/2758

## 6. Recommendations for brepkit

### Priority 1: Split kernel.rs (Low Risk, High Impact)

The current `kernel.rs` is ~7800 lines with 187 `pub fn` methods. This is
the single highest-maintenance-cost file in the project.

**Recommended split** (following truck-js's pattern):

```
crates/wasm/src/
  kernel.rs          # BrepKernel struct, constructor, core state management (~200 lines)
  primitives.rs      # makeBox, makeCylinder, makeSphere, etc.
  booleans.rs        # fuse, cut, intersect, compoundCut
  transforms.rs      # transform, mirror, copy
  measure.rs         # volume, surfaceArea, boundingBox, centerOfMass
  tessellate.rs      # tessellateSolid, tessellateSolidGrouped, wireframe
  sketch.rs          # sketch constraint solver bindings
  query.rs           # edge/face/vertex queries, topology introspection
  io.rs              # (already exists) STEP/STL/3MF import/export
  batch.rs           # executeBatch + dispatch_op
  error.rs           # (already exists) WasmError, validation helpers
  shapes.rs          # (already exists) JsMesh, JsEdgeLines, JsPoint3
  operations.rs      # extrude, revolve, sweep, loft, fillet, chamfer, etc.
```

Each module would contain `#[wasm_bindgen] impl BrepKernel { ... }` blocks.
wasm-bindgen supports multiple impl blocks across different modules for the
same struct.

### Priority 2: Fix catch_unwind Usage (High Risk)

Lines 641 and 6457 in kernel.rs use `catch_unwind` which is a no-op on
wasm32 with `panic=abort`. Replace with:

```rust
// Instead of catch_unwind, validate inputs exhaustively BEFORE the operation
fn fillet_safe(
    topo: &mut Topology,
    solid_id: SolidId,
    edge_ids: &[EdgeId],
    radius: f64,
) -> Result<SolidId, WasmError> {
    // Pre-validate: all edges exist, radius is positive, edges are valid for fillet
    // Then call the operation (which should never panic with valid inputs)
    try_fillet(topo, solid_id, edge_ids, radius)
        .map_err(WasmError::from)
}
```

### Priority 3: Batch Dispatch Code Generation (Medium Impact)

The `dispatch_op` function (~1700 lines) has massive boilerplate: every
operation repeats the same pattern of extracting JSON args, calling the
operation, and converting the result.

**Option A: Declarative macro** (simplest, no new crate needed):

```rust
macro_rules! batch_ops {
    (
        $self:ident, $op:ident, $args:ident;
        $(
            $name:literal => {
                $( let $param:ident : $ty:tt = $extract:expr; )*
                $body:expr
            }
        )*
    ) => {
        match $op {
            $(
                $name => {
                    $( let $param: $ty = $extract; )*
                    let result = $body;
                    result.map(|v| serde_json::json!(v)).map_err(|e: String| e)
                }
            )*
            _ => Err(format!("unknown operation: {}", $op))
        }
    };
}
```

**Option B: Proc macro** (more powerful, new crate needed):

A `#[batch_op]` attribute macro that generates both the direct wasm-bindgen
method and the dispatch arm from the same function definition. This is the
ideal long-term solution but requires a `brepkit-wasm-macros` crate.

**Option C: Build script code generation** (OpenCascade.js approach adapted):

A `build.rs` that reads operation metadata from a TOML/YAML file and
generates both the wasm-bindgen methods and the dispatch table. This avoids
proc macro complexity but adds a build-time dependency.

### Priority 4: Typed Batch Results via serde-wasm-bindgen

Replace JSON string batch results with structured data:

```rust
// Current: returns JSON string, JS must parse
pub fn execute_batch(&mut self, json: &str) -> String { ... }

// Better: return JsValue directly via serde-wasm-bindgen
pub fn execute_batch(&mut self, ops: JsValue) -> Result<JsValue, JsError> {
    let ops: Vec<BatchOp> = serde_wasm_bindgen::from_value(ops)?;
    let results: Vec<BatchResult> = ops.iter().map(|op| self.dispatch(op)).collect();
    serde_wasm_bindgen::to_value(&results).map_err(|e| JsError::new(&e.to_string()))
}
```

This eliminates the JSON serialize/parse round-trip and gives JS callers
typed results. Use `tsify-next` to generate TypeScript definitions for
`BatchOp` and `BatchResult`.

### Priority 5: Packed Buffer Pattern (Already Good)

brepkit's `JsMesh::packed_buffer()` is already following the optimal pattern.
This is validated by Manifold's approach (single typed_memory_view + slice).

**One improvement:** The current `positions()`, `normals()`, `indices()`
getters each `.clone()` the entire Vec. Consider marking them as returning
references to WASM memory (using `wasm_bindgen(getter)` with `js_sys::Float64Array`):

```rust
#[wasm_bindgen(getter)]
pub fn positions(&self) -> js_sys::Float64Array {
    // Create a view into WASM memory -- zero-copy
    unsafe { js_sys::Float64Array::view(&self.positions) }
}
```

**Caveat:** The view is invalidated by any WASM memory growth. The JS caller
must copy the data before any other WASM call. This is the same pattern
Manifold uses. Document this clearly.

### Not Recommended

- **`wasm-bindgen-struct`**: Designed for JS imports, not Rust exports.
- **`wasm_refgen`**: Designed for JS-to-Rust type upcasting, not relevant.
- **`linkme`/`inventory`**: Do not work on wasm32-unknown-unknown.
- **`AbstractShape` enum pattern** (from truck): Unnecessary with brepkit's
  handle-based architecture. Adding a JS-visible enum wrapper would increase
  API surface without benefit.
- **Emscripten-style binding generation** (from OpenCascade.js): Python-based
  code generation is not idiomatic in a Rust project.

## Confidence Levels

| Finding | Confidence | Reasoning |
|---------|-----------|-----------|
| `catch_unwind` is no-op on wasm32 panic=abort | **High** | Documented behavior, confirmed by multiple sources |
| linkme/inventory don't work on wasm32 | **High** | Maintainer closed linkme WASM issue; inventory requires global ctors |
| Splitting kernel.rs across modules works with wasm-bindgen | **High** | Multiple impl blocks for same struct supported, truck-js uses this |
| tsify-next is the maintained fork | **High** | Original tsify unmaintained, tsify-next actively developed |
| Proc macro composing with wasm-bindgen works | **Medium** | Proc macros can emit other proc macro attributes in theory, but no large-scale example found |
| serde-wasm-bindgen faster than JSON for batch results | **Medium** | Benchmarks vary 0.5x-2x depending on data shape and runtime |
| `Float64Array::view` zero-copy pattern | **Medium** | Works but invalidated by WASM memory growth; needs careful API documentation |
